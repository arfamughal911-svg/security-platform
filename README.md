# ThreatScope

**AI-Assisted Cyber Incident Reporting and Threat Alert System**

## Overview

ThreatScope is a distributed cybersecurity platform designed to automate the complete lifecycle of threat intelligence processing. The system continuously ingests real-time threat data from external sources, extracts Indicators of Compromise (IOCs) such as malicious IP addresses and suspicious domain names, evaluates each IOC using a severity scoring mechanism, and persists enriched records to a relational database.

All functionality is exposed via RESTful APIs and a real-time web dashboard, enabling both programmatic access and interactive monitoring.

The platform addresses a critical operational challenge in modern cybersecurity: the overwhelming volume of IOC data generated by threat intelligence feeds. Manual processing is no longer feasible at scale. ThreatScope mitigates this problem through automation, asynchronous messaging, and a well-structured microservices architecture.

---

## Architecture

ThreatScope follows a microservices-inspired modular architecture implemented within a Spring Boot application. Each service is organized into its own package, consisting of:

* A **Service class** for business logic
* A **Controller class** for REST API exposure

Communication patterns include:

* **Asynchronous communication** via Apache Kafka for IOC processing
* **Synchronous communication** via Spring dependency injection for internal coordination

### Core Services

**Ingestion Service**
Responsible for retrieving raw threat data from external APIs (AbuseIPDB and AlienVault OTX). Implements a retry mechanism with up to three attempts and a two-second delay between retries.

**Extraction Service**
Parses raw JSON responses and converts them into structured IOC objects. Each IOC includes:

* Type (IP or Domain)
* Value (address or hostname)
* Source (originating platform)

Parsing failures are handled gracefully to ensure pipeline continuity.

**Kafka Producer Service**
Serializes IOC objects into a colon-delimited format (`TYPE:VALUE:SOURCE`) and publishes them to the Kafka topic `ioc-topic` using Spring Kafka.

**Kafka Consumer Service**
Consumes messages from `ioc-topic` under the `security-group` consumer group. Messages are forwarded to the Ranking Service. Configured with `auto-offset-reset=earliest` to prevent data loss on restart.

**Ranking Service**
Evaluates IOC severity:

* IPs are scored via AbuseIPDB’s `/check` endpoint (0–100 scale)
* Domains are assigned a default score of 0

Includes retry logic (three attempts, one-second delay). Outputs enriched IOC entities.

**Database Service**
Manages persistence using Spring Data JPA.

* Checks for duplicates via `findByValue()`
* Stores only unique records
* Uses Hibernate for ORM and schema management

## External API Integration

**AbuseIPDB**

* `/api/v2/blacklist`: Retrieves blacklisted IP addresses
* `/api/v2/check`: Provides abuse confidence score for IPs
* Authentication via `Key` header

**AlienVault OTX**

* `/api/v1/pulses/subscribed`: Retrieves recent threat pulses
* Extracts indicators (IPv4, domain, hostname)
* Filters to IP and domain types only
* Authentication via `X-OTX-API-KEY` header

## Database Design

The system uses a MySQL database named `securitydb` with a primary table `iocs`.

| Column         | Data Type | Constraints        | Description              |
| -------------- | --------- | ------------------ | ------------------------ |
| id             | BIGINT    | PK, AUTO_INCREMENT | Unique identifier        |
| type           | VARCHAR   | NOT NULL           | IOC type (IP or Domain)  |
| value          | VARCHAR   | NOT NULL           | IOC value                |
| source         | VARCHAR   | NOT NULL           | Data source              |
| severity_score | INT       | DEFAULT 0          | Risk score (0–100)       |
| timestamp      | DATETIME  | NOT NULL           | Auto-generated on insert |

**Repository Features**

* `findByValue()` → deduplication
* `findBySeverityScoreGreaterThan()` → high-risk filtering
* `findByType()` → type filtering
* `findBySource()` → source filtering

## Data Flow

1. Ingestion Service retrieves raw JSON from external APIs
2. Extraction Service converts raw data into structured IOC objects
3. Kafka Producer publishes IOCs to `ioc-topic`
4. Kafka Consumer receives and forwards messages
5. Ranking Service assigns severity scores
6. Database Service deduplicates and stores records
7. Data becomes accessible via APIs and dashboard
