# ThreatScope

AI-Assisted Cyber Incident Reporting and Threat Alert System

## Overview

ThreatScope is a distributed cybersecurity platform designed to automate the full
lifecycle of threat intelligence processing. The system continuously ingests live
threat data from external sources, extracts Indicators of Compromise (IOCs) such
as malicious IP addresses and suspicious domain names, evaluates each IOC using a
severity scoring mechanism, and persists enriched records to a relational database.
All functionality is exposed through RESTful APIs and a real-time web dashboard.

The platform was built to address a genuine operational problem in cybersecurity:
the volume of IOC data generated by modern threat feeds makes manual processing
impractical. ThreatScope solves this through automation, asynchronous messaging,
and a clean microservices architecture.

---

## Architecture

The platform follows a microservices architecture implemented as a modular Spring
Boot application. Each logical service resides in its own Java package with a
dedicated Service class containing business logic and a Controller class exposing
REST endpoints. Services communicate asynchronously via Apache Kafka for the core
IOC processing pipeline, and synchronously via Spring bean injection for
coordinating ingestion and extraction within the same request.

The six core services are:

**Ingestion Service**
Connects to AbuseIPDB and AlienVault OTX via HTTP and retrieves raw JSON threat
data. Implements a retry mechanism with three attempts and a two-second delay
between retries before returning a failure response.

**Extraction Service**
Parses raw JSON responses from both external sources and constructs structured IOC
objects. Each IOC carries three fields: type (IP or Domain), value (the actual
address or hostname), and source (the originating platform). Parsing exceptions
are handled gracefully without halting the pipeline.

**Kafka Producer Service**
Serialises each IOC object as a colon-delimited string in the format
TYPE:VALUE:SOURCE and publishes it to the Kafka topic ioc-topic using Spring's
KafkaTemplate.

**Kafka Consumer Service**
Listens on ioc-topic using a @KafkaListener annotation under the consumer group
security-group. Each consumed message is forwarded to the Ranking Service for
processing. Configured with auto-offset-reset set to earliest to ensure no
messages are skipped on restart.

**Ranking Service**
Queries the AbuseIPDB /check endpoint for each IP IOC and retrieves an
abuseConfidenceScore between 0 and 100. Domain IOCs are assigned a default score
of 0 as AbuseIPDB does not support domain scoring. Implements a three-attempt
retry loop with a one-second delay. Constructs an IOCEntity object and passes it
to the Database Service.

**Database Service**
Handles all MySQL interactions through the IOCRepository interface. Before
persisting any record, calls findByValue() to check for duplicates. Duplicate
entries are silently discarded. Unique records are saved via JPA's save() method.

## Technology Stack

**Language:** Java 17

**Framework:** Spring Boot 3 built on spring-boot-starter-parent. The application
uses a modular monolith approach where each microservice package contains its own
Service and Controller classes. REST endpoints are exposed using Spring MVC
annotations.

**Message Broker:** Apache Kafka integrated via the spring-kafka library. IOCs are
published to and consumed from a single topic named ioc-topic. The producer
serialises each IOC as a TYPE:VALUE:SOURCE string. The consumer deserialises and
routes messages to the Ranking Service asynchronously.

**Database:** MySQL accessed through Spring Data JPA and Hibernate ORM. The schema
is automatically generated using Hibernate's DDL auto update setting, which
creates the iocs table on first startup and applies structural changes on
subsequent runs without dropping existing data.

**HTTP Client:** Spring RestTemplate used for all outbound HTTP communication with
external APIs. API keys are injected from the application configuration file using
the @Value annotation and are never hardcoded in source logic.

**JSON Parsing:** The org.json library is used to parse JSONObject and JSONArray
structures returned by both external APIs.

**Frontend:** A static HTML dashboard served directly by Spring Boot. Built using
HTML, CSS, and vanilla JavaScript. The dashboard fetches live data from the backend
using the JavaScript Fetch API.

**Build Tool:** Apache Maven for dependency management via pom.xml.

## External API Integration

**AbuseIPDB**
Two endpoints are used. The /api/v2/blacklist endpoint is called by the Ingestion
Service with a limit of 20 and returns a JSON array of blacklisted IP addresses.
The /api/v2/check endpoint is called by the Ranking Service for each IP IOC and
returns the abuseConfidenceScore value used as the severity score. Both calls
require the API key in the Key HTTP request header.

**AlienVault OTX**
The /api/v1/pulses/subscribed endpoint is called with a limit of 5, returning the
most recent threat pulses. Each pulse contains an indicators array where each entry
has a type field (IPv4, domain, or hostname) and an indicator field with the actual
value. The Extraction Service filters these to retain only IP and domain types. The
API key is supplied in the X-OTX-API-KEY HTTP request header.

## Database Design

The platform uses a single MySQL database named securitydb containing one primary
table, iocs. The schema is maintained automatically by Hibernate.

| Column         | Data Type | Constraints              | Description                          |
|----------------|-----------|--------------------------|--------------------------------------|
| id             | BIGINT    | PRIMARY KEY, AUTO_INCREMENT | Unique identifier for each record |
| type           | VARCHAR   | NOT NULL                 | IOC category: IP or Domain           |
| value          | VARCHAR   | NOT NULL                 | The IP address or domain name        |
| source         | VARCHAR   | NOT NULL                 | Origin: AbuseIPDB or AlienVault      |
| severity_score | INT       | DEFAULT 0                | Abuse confidence score from 0 to 100 |
| timestamp      | DATETIME  | NOT NULL                 | Auto-set via @PrePersist hook        |

Deduplication is enforced at the application layer. The IOCRepository interface
extends JpaRepository and declares four custom query methods:
findBySeverityScoreGreaterThan for high-risk filtering, findByType and findBySource
for category-based filtering, and findByValue for the deduplication check.

## REST API Endpoints

| Method | Endpoint                      | Description                                      |
|--------|-------------------------------|--------------------------------------------------|
| GET    | /api/ingest/abuseipdb         | Fetch raw JSON from AbuseIPDB blacklist          |
| GET    | /api/ingest/alienvault        | Fetch raw JSON from AlienVault OTX               |
| GET    | /api/ingest/all               | Fetch from both sources                          |
| GET    | /api/extract/abuseipdb        | Return extracted IOC list from AbuseIPDB         |
| GET    | /api/extract/alienvault       | Return extracted IOC list from AlienVault        |
| GET    | /api/extract/all              | Return combined IOC list from both sources       |
| POST   | /api/kafka/publish/abuseipdb  | Publish AbuseIPDB IOCs to Kafka topic            |
| POST   | /api/kafka/publish/alienvault | Publish AlienVault IOCs to Kafka topic           |
| POST   | /api/kafka/publish/all        | Publish all IOCs from both sources to Kafka      |
| GET    | /api/rank/check               | Score a specific IP on demand via query param    |
| GET    | /api/iocs                     | Return all stored IOC records                    |
| GET    | /api/iocs/high-risk           | Return IOCs with severity score greater than 75  |
| GET    | /api/iocs/type/{type}         | Filter IOCs by type: IP or Domain                |
| GET    | /api/iocs/source/{source}     | Filter IOCs by source: AbuseIPDB or AlienVault   |
| GET    | /api/iocs/stats               | Return summary statistics as JSON                |

## Dashboard

ThreatScope includes a web-based monitoring dashboard served at /dashboard.html.
The dashboard displays real-time statistics including total IOC count, high-risk
alert count, IP address count, and domain count. It provides a filterable and
searchable table of all IOC records with colour-coded severity badges distinguishing
high-risk, medium-risk, and low-risk entries. Action buttons allow users to trigger
the full data pipeline directly from the browser by calling the Kafka publish
endpoints.

## Data Flow

1. A REST endpoint on the Ingestion Service is called, fetching raw JSON from
   AbuseIPDB and AlienVault OTX.
2. The Extraction Service parses the JSON and constructs a list of IOC objects,
   each with a type, value, and source.
3. The Kafka Controller iterates over extracted IOCs and passes each to the Kafka
   Producer, which serialises and publishes it to ioc-topic.
4. The Kafka Consumer asynchronously receives each message from the topic.
5. The Ranking Service is invoked for each message. IP IOCs are scored via
   AbuseIPDB. Domain IOCs receive a default score of 0.
6. An IOCEntity is constructed with all enriched fields and passed to the Database
   Service.
7. The Database Service checks for duplicates and persists new records to MySQL.
   Records are then available through the REST API and the ThreatScope dashboard.
